### Words we will use

A collection is a container for a group of objects, such as a list, stack, tree, or graph.

## Description

The iterator pattern lets you traverse <span style={{color: "var(--primary-font-color)"}}>a</span> collection without knowing its traversal implementation <span style={{color: "var(--writer-additional-note-font-color)"}}>[[ internal method for getting a single item and the algorithm for tracking which items have already been visited ]]</span>

## Problem without the pattern

When a client wants to traverse a collection,

the client needs to
1. know which method they can use to get a single item from the collection 
1. keep track of which items they have seen

But each collection provides a different method for accessing its items.

For example, an array might provide `array.getItem`, while a graph might provide `graph.getNode`.

Different access methods across collections force the client to know each collection's specific API.

It also adds more complexity when the client has to track visited items, which can lead to duplicated code each time they traverse a collection.

## With pattern

The iterator pattern introduces an `iterator` object that contains the traversal algorithm and the methods required for iteration.
```typescript title= showLineNumbers
interface Iterator<T> {
  getNextItem(): T;
  hasMoreItem(): boolean;
}
```
`getNextItem()` returns an item from the collection.

`hasMoreItem()` checks whether all items have been traversed.


The collection then needs to return an `iterator` object to the client by implementing an additional method such as `collection.getIterator`.

<Accordion title="Collection implementation example">

```typescript title=FooCollection.ts showLineNumbers
class FooCollection {
  // simplified
  add(){}
  getItem(){}
  size(){}

  getIterator(): Iterator<number> {
    return new EvenItemIterator(this);
  }
}
```

```typescript title=EvenItemIterator.ts showLineNumbers
class EvenItemIterator implements Iterator<number> {
  private index = 0;

  // traverse behavior is coupled with a collection
  constructor(private collection: FooCollection) {}

  getNextItem(): number {
    if (this.hasMoreItem()) {
      const result = this.collection.getItem(this.index);
      this.index += 2
      return result
    }
    
    throw new Error('no item left')
  }

  hasMoreItem(): boolean {
    return this.index < this.collection.size();
  }
}
```

</Accordion>


The client then uses the `iterator` object to iterate.

```typescript title= showLineNumbers
const iterator: Iterator = collection.getIterator()

while (iterator.hasMoreItem()){
  const item  = iterator.getNextItem()
  // do something with item
}
```

With the `iterator` interface, the client does not need to know which method each collection uses to access its items. It can just use `iterator.getNextItem()`.

The traversal algorithm and visited-item tracking are pushed into the collection's responsibility. The client can simply call `iterator.hasMoreItem()`.



<Note> 

Client still need to know which method of the collection is used to return `iterator`

Some designs have all collections implement the same interface for returning an `iterator`. For example:
```typescript title= showLineNumbers
interface IterableCollection<T>{
  getIterator(): Iterator<T>
}
```

Then, regardless of which collection the client is dealing with, it can just call `collection.getIterator()`.

But returning just one `iterator` might not be enough for some collections.

For example, a tree collection may need both depth-first and breadth-first traversal algorithms, so it needs more than one iterator method.
```typescript title= showLineNumbers
interface Tree{
  // ...
  getDepthFirstIterator(): Iterator;
  getBreadthFirstIterator(): Iterator;
}
```

In this case, the client still needs to know the methods available on a `Tree`.

<Info> 

Different kinds of `tree` can implement the same `Tree` interface.

</Info>

</Note>

<Tip> 

See [diagrams](https://refactoring.guru/design-patterns/iterator#structure) for better understanding

</Tip>

<Info> 

One iterator corresponds to one traversal behavior.

</Info>

<Info> 

Several iterators can traverse the same collection at the same time.

</Info>


## When to use

- create a custom data structure
- adding or reusing traversal behavior
- dealing with streams of data under memory constraints
- lazily computing items in a collection
- eg. computing all items at once consumes resources. With `iterator.getNextItem()`, you compute one item at a time
- delay an iteration and continue when needed
