This page shows how to take full advantage of special documentation capabilities in OpenAPI like markdown syntax or example objects.

## Documentation Fields

&#8203;<span style={{color: "var(--secondary-font-color)"}}> Almost </span> every object in the OpenAPI Specification accepts a `description` field <span style={{color: "var(--not-require-but-good-to-know-font-color)"}}> which can provide additional information for developers </span>

For instance, a parameter's `name`, `type` and valid range of values are already present in the API definition.
The `description` field can complement this information by explaining the purpose of this parameter, the effect of each value or possible interactions with other parameters:

``` yaml lines highlight={12-17}
paths:
  /audio/volume:
    put:
      requestBody:
        required: true
        content:
          application/json:
            schema:
              type: integer
              minimum: 0
              maximum: 11
              description:
                Current volume for all audio output.
                0 means no audio output (mute). 10 is the maximum value. 11 enables
                the overdrive system (danger!).
                When set to 0 all other audio settings have no effect.
              ...
```

<Tip>

Additionally, the [Path Item](https://spec.openapis.org/oas/v3.1.0#path-item-object), [Operation](https://spec.openapis.org/oas/v3.1.0#operation-object) and [Example objects](https://spec.openapis.org/oas/v3.1.0#example-object), which usually have long descriptions, accept a `summary` field <span style={{color: "var(--secondary-font-color)"}}> providing a short description </span>.

</Tip>

## Providing Long Descriptions in YAML

### Plain Mode


``` yaml lines
description: 
  Current volume for all audio output.
  0 means no audio output (mute). 10 is the maximum value. 11 enables
  the overdrive system (danger!).
  When set to 0 all other audio settings have no effect.
  ...
```

The above example uses the <span style={{color: "var(--secondary-font-color)"}}> plain mode </span> for strings. It is easy to use because it does not require any special syntax( `|` or `>`).

But it can confuse YAML-parsing tools if the string contains colon `:` or hash `#` characters.

In these cases, the whole string must be enclosed in single or double quotes:

``` yaml
description: "Beware of lines containing colons and hashes like this: #"
```

### Literal Mode

Literal mode use Pipe `|` as an indicator character.

Line breaks in the source YAML file <span style={{color: "var(--secondary-font-color)"}}> are preserved in the output </span>.

``` yaml lines
description: |
  This is a string
  in multiple lines.

  And an extra one.
```

Output:

``` txt lines
This is a string
in multiple lines.

And an extra one.
```

### Folded Mode

Folded mode use Greater-than sign `>` as an indicator character.

&#8203;<span style={{color: "var(--secondary-font-color)"}}> Line breaks are removed so the output is a single string. </span>

``` yaml lines
description: >
  This is a string
  in multiple lines.

  And an extra one.
```

Output:

``` yaml lines
This is a string in multiple lines.
And an extra one.
```

<Note>

Use an empty line to force a line break.

</Note>

<Info>

The literal and folded modes do not need to use quotes.

</Info>

## CommonMark Syntax

`description` fields allow rich text formatting by using [CommonMark 0.27](https://spec.commonmark.org/0.27/).

<span style={{color: "var(--not-require-but-good-to-know-font-color)"}}>

Far more features than listed below are available (including HTML tags),
but given that OpenAPI descriptions are meant to be included inside larger auto-generated documentation,
using more advanced formatting is typically very complicated.

</span>

### Headings

``` markdown lines
# Level 1

## Level 2

### Level 3
```

### Emphasis


``` markdown lines
*Emphasis*

**Strong Emphasis**

***Both***
```

### Lists

``` markdown lines
- Item 1
- Item 2
  - Item 2.1
```

### Code

``` `markdown lines
An inline `code span`.

```
A fenced code block
```
````

### Links

```markdown title= showLineNumbers
[Link text](Link URL)
![Alt text](Image URL)
```

## The Example Object

Some OpenAPI objects can list examples explicitly instead of having them embedded in the description field, <span style={{color: "var(--not-require-but-good-to-know-font-color)"}}> enabling automated processing by tools </span>.

This allows, among other things:

- Special rendering of the examples inside the documentation.
- [Example objects](https://spec.openapis.org/oas/v3.1.0#example-object) can be used by mock servers as return values.

Two different fields provide this functionality:

- `example` allows one sample
- `examples` allows multiple.

<Note>

<span style={{color: "var(--secondary-font-color)"}}> Only one </span> of the two fields can
be present in each object.

</Note>

### `example` field

The content of the `example` field (found in Parameter, Media Type and Schema Objects) must match the format of its parent object:

``` yaml lines highlight={6}
schema:
  coordinate:
    type: integer
    minimum: 1
    maximum: 3
    example: 1
```

### `example`<span style={{color: "var(--primary-font-color)"}}>`s`</span> field

The `examples` field (found in Parameter and Media Type Objects) is a map pairing an example name with an [Example Object](https://spec.openapis.org/oas/v3.1.0#example-object).

The Example Object can also provide a `summary` and a `description` field along with its value ( `value` or `externalValue` field ).

``` yaml lines highlight={8-14}
responses:
  "400":
    description: The provided parameters are incorrect
    content:
      text/html: # This is a Media Type Object
        schema:
          type: string
        examples:
          illegalCoordinates:
            value: "Illegal coordinates."
          notEmpty:
            value: "Square is not empty."
          invalidMark:
            value: "Invalid Mark (X or O)."
```
