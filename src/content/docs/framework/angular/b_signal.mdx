# Signals

A signal is a value that notifies its consumers when it changes.

## Create signal

```typescript showLineNumbers
import { signal } from "@angular/core";
const firstName: WritableSignal<string> = signal("Morgan");
```

## Read signal value

Call its getter function.

```typescript showLineNumbers
console.log(firstName());
```

## Update signal value

```typescript showLineNumbers
firstName.set("Jaime");
firstName.update((name) => name.toUpperCase());
```

<Aside type="note" title="info">

A writable signal has the [`WritableSignal`](https://angular.dev/api/core/WritableSignal) type.

</Aside>

## Computed signal

[`computed`](https://angular.dev/api/core/computed) is a <Color color="var(--secondary-font-color)">read-only</Color> `signal` that derives its value from other signals.

```typescript showLineNumbers
import { signal, computed } from "@angular/core";

const firstName = signal("Morgan");
const firstNameCapitalized = computed(() => firstName().toUpperCase());
```

The callback function passed to `computed` runs whenever a parent signal changes.

<Aside type="note" title="note">

`computed` signals are lazily evaluated and cached.

The callback passed to `computed` does not run until the computed signal is read for the first time, for example `firstNameCapitalized()`.

Angular invalidates the cache when a parent signal value changes.

</Aside>

<Aside type="note" title="info">

A `computed` signal has no `update` or `set` method.

</Aside>

<Aside type="note" title="info">

`computed` signal has [`Signal`](https://angular.dev/api/core/Signal) type

</Aside>

## Effect

An effect is a code block that executes whenever one of its parent signals changes.

```typescript showLineNumbers
import { effect } from "@angular/core";

effect(() => {
  console.log(
    `log on initialization and later when the count signal changes ${count()}`
  );
});
```

For `effect` to know its parent signals, Angular executes the effect block at definition time; it is not lazy.

Effects can only be created inside an injection context.

<details>

<summary> example </summary>

```typescript showLineNumbers
@Component({...})
export class MyComponent {
  // highlight-start
  private barEffect =  effect(() => {
    console.log('bar effect', fooSignal())
  });
  // highlight-end

  constructor() {
    // highlight-start
    effect(() => {
      console.log('something with', fooSignal())
    });
    // highlight-end
  }
}
```

</details>

To create an `effect` outside an injection context, pass an `injector` to it.

```typescript showLineNumbers
@Component({...})
export class EffectiveCounterComponent {
  readonly count = signal(0);
  private injector = inject(Injector);

  initializeLogging(): void {
    effect(() => {
      console.log(`The count is: ${this.count()}`);
      // highlight-next-line
    }, {injector: this.injector});
  }
}
```

<Aside type="note" title="info">

Effects always execute asynchronously during the change detection process.

</Aside>

### Destroying effect

`effect` is automatically destroyed when its enclosing context is destroyed.

So, an effect that lives inside a component, directive, or service is destroyed whenever the component, directive, or service is destroyed.

<Aside type="note" title="note">

An `effect` is also destroyed before it is rerun.

</Aside>

### Manually destroying effect

Use [`effect.destroy()`](https://angular.dev/api/core/EffectRef#destroy).

```typescript showLineNumbers
const effectRef = effect(() => {});
effectRef.destroy();
```

### Disable automatically destroyed

An `effect` can be configured to last until it is manually destroyed by using the `manualCleanup` option.

```typescript showLineNumbers
effect(() => {}, { manualCleanup: true });
```

### Effect cleanup function

You can execute a function when the `effect` is destroyed.

```typescript showLineNumbers
// highlight-next-line
effect((onCleanup) => {
  const timer = setTimeout(() => {}, 1000);

  // highlight-start
  onCleanup(() => {
    clearTimeout(timer);
  });
  // highlight-end
});
```

## Dependencies tracking

Angular uses the term "dependencies" for parent signals.

<details>

<summary> example </summary>

`bar` is a `computed` value or `effect` that reads the `foo` signal, so `foo` is a dependency of `bar`.

```typescript showLineNumbers
const bar = effect(() => foo());
// or
const bar = computed(() => foo());
```

</details>

Dependencies are the signals that are read during child code block execution, regardless of where those signals are defined.

Dependencies are tracked to determine whether the child code block should be re-executed.

If dependencies change, the child code block is re-executed.

### Example: reading outside of child code block

```typescript showLineNumbers
const isFoo = signal(false);
const bar = signal("bar");

effect(() => {
  isFoo();
  external();
});

function external() {
  console.log(bar());
}
```

`bar` is still being tracked even though it is not <Color color="var(--secondary-font-color)">directly read</Color> inside the child code block.

### Example: conditional reading

```typescript showLineNumbers
const isFoo = signal(false);
const bar = signal("bar");

const baz = computed(() => {
  if (isFoo()) {
    return bar();
  } else {
    return "nothing";
  }
});
```

On the first evaluation of `baz`, `isFoo` is `false`, so the `bar` signal is not read.

Therefore, only changes to `isFoo` trigger the `baz` code block, not changes to `bar`.

Only after `isFoo` changes to `true`, causing `bar` to be read, do both `isFoo` and `bar` trigger the `baz` code block.

<Aside type="note" title="note">

Dependency changes do not always cause the child value to change.

They do, however, cause the child code block to re-execute, even if the new value ends up being the same as the previous one.

</Aside>

### Reading without tracking dependencies

You can prevent a signal from being tracked by using [`untracked`](https://angular.dev/api/core/untracked).

```typescript showLineNumbers
effect(() => {
  const barValue = untracked(barSignal);
  console.log(fooSignal(), barValue);
});
```

```typescript showLineNumbers
// untracked accept callback function
effect(() => {
  fooSignal();

  untracked(() => {
    // barSignal is not tracked
    external();
  });
});

function external() {
  barSignal();
}
```

This makes the `effect` execute only when `fooSignal` emits a change.

Without `untracked`, the `effect` would execute whenever `fooSignal` <Color color="var(--secondary-font-color)">or</Color> `barSignal` emits a change.

## Linked signal

[`linkedSignal`](https://angular.dev/api/core/linkedSignal) is like `computed`, but it is writable.

```typescript showLineNumbers
import { linkedSignal, signal } from "@angular/core";

const foo = signal("foo");
const myLinkedSignal = linkedSignal(() => {
  return foo() + " computed";
});

myLinkedSignal();
// foo computed

myLinkedSignal.set("bar");
myLinkedSignal();
// bar

foo.set("new foo");
myLinkedSignal();
// new foo computed
```

It recomputes its value when dependencies emit changes.

It can also update its own value.

<Aside type="note" title="note">

`linkedSignal` is also lazily evaluated and cached.

</Aside>

### Accessing previous value

`linkedSignal` allows you to access its previous value and the previous values of its dependencies.

```typescript showLineNumbers
const foo = signal(123);

linkedSignal({
  source: foo,
  computation: (fooNewValue, previous) => {
    const linkedSignalPreviousValue = previous?.value;
    const fooPreviousValue = previous?.source;
    return "something" + fooNewValue;
  },
});
```

`computation` is the callback function that produces the value for `linkedSignal`. It plays the same role as the callback in a normal `linkedSignal`.

`source` is typically a dependency of `linkedSignal`. It is a function that is called, and its return value is passed as the first argument to the `computation` callback (`fooNewValue`).

<details>

<summary> example </summary>

```typescript showLineNumbers
linkedSignal({
  source: () => 123,
  computation: (newSourceValue, previous) => {
    console.log(newSourceValue);
    // 123
  },
});
```

</details>

The second argument of the `computation` function, `previous`, is an object containing two keys:

- `previous?.value` is the previous return value of the `computation` function (the previous value of `linkedSignal`)
- `previous?.source` is the previous return value of the `source` function

<Aside type="note" title="note">

Any signals called from either the `source` or `computation` function are tracked.

</Aside>

<Aside type="note" title="note">

It is necessary to provide type arguments to `linkedSignal`.

```typescript showLineNumbers
linkedSignal<SourceReturnedValue, ComputationReturnValue>(...)
```

</Aside>

<Aside type="note" title="note">

`previous` could be `undefined`

</Aside>

<Aside type="tip" title="tip">

By making `source` return an object instead of a single value, you can access multiple source values from the previous cycle.

```typescript showLineNumbers
linkedSignal({
  source: () => {key1: 1 , key2: 2},
  computation: (newSourceValue, previous) => {
    console.log(previous?.source.key1, previous?.source.key2)
  },
});
```

</Aside>

## Signal equality functions

You can provide a function to determine whether a signal value has changed.

```typescript showLineNumbers
import _ from "lodash";

const sample = {key1: 1, key2: 2}

signal(sample, { equal: _isEqual });
signal(sample, { equal: (previousValue, newValue) => previousValue.key1 === newValue.key1);

// linkedSignal
linkedSignal(() => 'foo', { equal: (previousValue, newValue) => previousValue.key1 === newValue.key1);
```

<Aside type="note" title="note">

The default signal equality function is [`Object.is`](https://developer.mozilla.org/docs/Web/JavaScript/Reference/Global_Objects/Object/is).

</Aside>

<Aside type="note" title="info">

It is used to determine whether the signal should trigger a change.

</Aside>
